# 从 Single-Agent OS 到 Constitutional Runtime

## 第一章：Agent OS 的结构性失稳

### 为什么长期 AI 工作需要任务归属、权限、审计与合法结案

一开始，我的 AI 系统很像一个围绕单一智能体搭建起来的操作系统。

它能读文件、写代码、调用工具、总结上下文、记住目标，也能在一个很长的对话里持续推进任务。最初，这个抽象看起来非常自然：一个 agent，一个注意力中心，一个不断演化的工作空间。用户提出目标，agent 负责理解、规划和执行。

但当任务越来越长、越来越复杂、越来越接近真实生产工作时，这个抽象开始出现裂缝。

问题不只是 agent 不够聪明。真正的问题是：系统把长期工作仍然当作一场对话来处理。

对话可以灵活，可以即兴，可以通过下一轮纠正来修复误解。但 runtime 不行。一个能长期工作的 AI 系统，不能只依靠“看起来还在理解”。它需要明确的任务归属，需要边界清晰的权限，需要可以追溯的审计记录，也需要一种合法地宣布“这件事已经完成”的机制。

这就是我决定重建系统的原因。

## Single-Agent OS 的幻觉

Single-Agent OS 最吸引人的地方，是它看起来非常完整。

用户意图、上下文、工具调用、计划、执行、复盘、总结，似乎都可以交给同一个 agent。对于短任务，这种模式非常有效。你让它改一个函数、解释一个报错、写一段文案、检查一个文件，它通常可以把上下文装进工作记忆里，然后很快完成。

短任务的容错空间很大。即使出错，用户也可以在下一轮对话里纠正。

但长期任务不是这样。

长期任务会跨越多个目标、多个文件、多个工具调用、多个中间决策，甚至多个工作会话。它会被打断，会被恢复，会出现需求变化，会需要委托，也会产生很多半完成状态。

这时，如果整个操作仍然由一个对话型 agent 隐式管理，系统就会出现一种危险的错觉：它看起来还在工作，但结构已经开始不稳定。

## 四个失稳点

我反复遇到的问题，最后集中在四个缺失的 runtime 原语上。

第一，缺少稳定的任务归属。

agent 可以说“我正在处理这个任务”，但系统并不一定真的知道这个任务是什么、谁拥有它、范围在哪里、进度怎么算、什么时候发生了所有权转移。长期任务如果没有归属，就会变成一团意图。它好像可以继续推进，却很难被可靠治理。

第二，权限太扁平。

很多 agent 系统里的工具权限像一个开关：能用，或者不能用。但真实工作里的权限并不是这样。读取文件和修改文件不是一回事；修改本地草稿和公开发布不是一回事；创建分支和推送公开仓库也不是一回事。一个 runtime 必须把权限理解为具体情境里的能力，而不是一句模糊的“我信任你”。

第三，审计太薄。

聊天记录不是审计记录。它可能包含线索，但它不是围绕责任设计的结构化历史。真正的审计需要回答：谁在什么时候基于什么输入做了什么决定，使用了什么权限，影响了什么 artifact，留下了什么不确定性。

第四，结案太随意。

agent 可以用一段很自信的总结结束工作。但自信不是结案。对于长期任务，“完成”不能只是“助手停止输出”。完成必须有证据：改了哪些文件，跑了哪些检查，哪些假设仍然存在，哪些风险还没有消除，用户需要知道什么。

这四个缺口，让 Single-Agent OS 在长期工作中天然不稳定。

## 让 agent 更聪明并不能解决一切

最直觉的解决方案，是换更强的模型。

更大的上下文、更好的推理、更强的工具使用能力、更长的记忆，这些当然都有帮助。但它们不能单独解决 runtime 问题。

一个更聪明的 agent，仍然可能在没有足够权限的情况下行动。

一个更大的上下文窗口，仍然可能把关键事件埋在海量文本里。

一个更好的记忆系统，如果没有来源和有效期，也可能只是更自信地记错。

更多工具如果没有权限边界，反而会扩大错误的影响范围。

所以问题不只是智能，而是治理。

人类组织早就学过这件事。严肃工作不是只靠聪明人完成的。它需要角色、权限、日志、复核、升级路径和完成定义。这些结构有时显得麻烦，但它们存在，是因为复杂性最终会击穿纯粹的临场协调。

AI 系统也正在走到这个阶段。

## Constitutional Runtime 的转向

我的结论不是放弃 agent，而是停止把 agent 当成整个操作系统。

在 Constitutional Runtime 里，agent 是一个运行在治理环境中的行动者。runtime 负责定义工作规则：

- 当前存在什么任务
- 谁拥有这个任务
- 这个任务有哪些权限
- 哪些动作必须确认
- 哪些事件必须记录
- 如何交接
- 什么证据足以支持结案

这会改变 AI 工作的形态。

agent 仍然可以创造、规划、写作、编码、检查和综合信息。但它不再需要把全部治理责任都塞进脆弱的对话状态里。宪法由 runtime 承载，agent 在宪法之内行动。

这就是我正在构建的系统基础。

## 第一章想说明什么

这一章不是在否定 single-agent 系统。恰恰相反，single-agent 系统非常有用。

我的判断更具体：当一个 single-agent OS 被要求管理长期、高后果、多阶段的工作，而它又没有显式的 runtime 治理机制时，它会发生结构性失稳。

如果我们希望 AI 系统能和我们一起工作几天、几周，甚至几个月，我们需要的不只是更好的 prompt。

我们需要任务归属。

我们需要权限边界。

我们需要审计记录。

我们需要合法结案。

当这些原语存在之后，系统就不再只是一个 assistant loop。

它开始成为一个 Constitutional Runtime。
